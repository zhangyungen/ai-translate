## Context
目标是从 0 到 1 构建 Chrome 翻译插件，并尽量对齐腾讯翻译插件交互形态。项目需要可持续迭代，因此在首次交付中引入清晰分层与可替换翻译网关。

对腾讯插件安装包的本地分析结论：
- 后台通过 `chrome.runtime.onMessage` 接收 `message_type=fetch`，统一转发到 `https://transmart.qq.com/api/imt`。
- 内容脚本在请求体 `header` 中补充 `client_key`。
- `client_key` 来源是本地派生而非短期 token：`("tencent_transmart_crx_" + btoa(navigator.userAgent)).slice(0,100)`。
- 语言能力列表 (`support_lang`) 做了本地缓存，过期策略约 24 小时后刷新。

## Goals / Non-Goals
- Goals:
  - 交付可安装、可用的 MV3 插件。
  - 支持任何语言互译，默认英语->中文体验最优先。
  - 提供整页翻译与阅读模式。
  - 明确异常与降级路径，异常路径输出 `warn` 日志。
- Non-Goals:
  - 不实现 OCR、语音翻译、文档上传翻译。
  - 不实现云端用户账户体系。

## Decisions
- Decision: 采用 DDD 分层并保持边界清晰。
  - Domain: `TranslationTask`、`LanguagePair`、`ReadingSegment`、`GlossaryItem`。
  - Application: 用例编排（整页翻译、划词翻译、阅读模式、hover 翻译）。
  - Infrastructure: `TransmartGateway`、本地缓存仓储、日志器、DOM 观察器。
  - Interface: popup UI 与 content UI。

- Decision: API 机制复用策略
  - 沿用腾讯插件 `client_key` 派生算法作为默认机制。
  - 提供 Key Provider 抽象，后续可替换为服务端签发机制。
  - 语言列表使用本地缓存 + TTL 自动续期（24h）。
  - 翻译文本结果采用双层缓存（运行时内存 + `chrome.storage.local` 持久层），键包含 `sourceLang/targetLang/text`。

- Decision: 整页翻译执行策略
  - 首次启用时仅翻译视口内文本节点。
  - 监听页面滚动/缩放与 DOM 变更，按可见区域持续增量翻译（看到哪里翻译到哪里）。
  - 使用元素位置+文本键做候选去重，避免动态渲染导致重复插入。
  - 对连续相同译文执行跳过策略，降低重复展示噪声。
  - 对“当前译文 == 前一条已展示主翻译结果”执行跳过策略。
  - 对“当前译文与当前位置上下文 5 行任一行一致”执行跳过策略。
  - 页眉/页脚候选除正文标签外，补充扫描 `header/nav/footer` 与语义 role 容器内的可见文本叶子节点，避免短文案漏翻。
  - 关闭整页翻译时按节点原文映射回滚。

- Decision: 页面级自动语言与无需翻译提示
  - 来源语言优先读取页面 `html[lang]`，缺失时对可见正文抽样检测。
  - 目标语言规则固定为：中文->英文，英文->中文，其他语言->浏览器默认语言。
  - 若来源语言与目标语言一致，则不执行翻译并展示“无需翻译”提示。

- Decision: 模式互斥与切换流程
  - `pageTranslateEnabled` 与 `readingModeEnabled` 必须互斥。
  - 切换到整页翻译：若阅读模式开启，先清理阅读模式产物，再执行整页翻译。
  - 切换到阅读模式：若整页翻译开启，先回滚整页翻译原文，再执行阅读模式翻译。
  - popup 通过当前 tab 状态接口展示已开启模式，避免用户误判当前状态。
  - popup 手动点击模式开关时，消息需携带 `sourceLang/targetLang`，content 侧优先应用该语言配置后再执行模式切换，确保“手动操作以弹窗语言为准”。

- Decision: 阅读模式分段
  - 采用增强规则分句（轻量 NLP）：句边界识别、缩写/小数保护、长句二次切分、短句合并。
  - 分段后使用 `  |  ` 拼接展示。
  - 后续可替换为更复杂的语义分段策略或服务端 NLP。

- Decision: 阅读模式展示序与去重
  - 展示顺序固定为：分割断句 -> 低频/生僻词翻译 -> 整段翻译。
  - 每次插入阅读模式产物前，按类型（断句/术语/整段）比较与对应“上一段已插入展示文本”是否一致；一致则跳过。
  - 术语行在单段内去重，避免相同词条重复展示。
  - 主翻译结果（术语/整段）维护“上一条已展示主翻译结果”，若相同则跳过。
  - 主翻译结果与当前位置上下文 5 行进行比对，若命中同文则跳过。
  - 使用“上一个已插入节点”为锚点执行 afterend，避免多次插入导致顺序反转。
  - 对少于 3 个单词的文本跳过断句展示，仅展示术语/整段翻译结果。
  - 断句块与整段翻译块采用不同背景色，降低视觉混淆。

- Decision: 代码区排除策略
  - 代码区识别采用标签与 class 关键词的向上查找（最大 6 层）。
  - 代码区默认不参与整页翻译、阅读模式候选与 hover 翻译。
  - 代码区选中翻译仅在整页翻译或阅读模式开启时放行。

- Alternatives considered:
  - 直接在 UI 层拼接 API 请求：实现快但耦合重，放弃。
  - 把所有翻译请求都从 content 直发网络：可行但不利于权限与统一治理，放弃。

## Risks / Trade-offs
- 腾讯接口稳定性风险：若限流或策略变化，可能影响可用性。
  - Mitigation: 限流重试、告警日志、可切换网关。
- 页面重排风险：整页替换可能破坏少量复杂站点布局。
  - Mitigation: 保留原文节点映射，支持回滚与禁用域名。
- 模式切换频繁时状态不一致风险（UI 显示与页面实际模式不一致）。
  - Mitigation: 统一以 tab 侧状态为准，popup 每次打开与每次切换后都刷新状态。
- 悬停翻译触发频率高：可能造成请求抖动。
  - Mitigation: 300ms 防抖 + 去重缓存。
- 滚动触发翻译可能导致高频扫描与请求压力。
  - Mitigation: 视口检测防抖、并发上限、翻译缓存命中优先。

## Migration Plan
1. 先完成基础插件骨架与 API 通路。
2. 完成 popup 翻译和整页翻译。
3. 增加阅读模式和术语预翻译。
4. 增加打包、测试与文档。

## Open Questions
- 低频词定义采用词频词典还是按长度/词性启发式，先按启发式实现。
- 腾讯接口字段若发生变化，是否允许通过配置覆盖，先保留配置入口。
